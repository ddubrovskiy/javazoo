# Moscow Zoo ERP System

### Дубровский Даниил Максимович

## Предметная область
Программа моделирует базовые процессы Московского зоопарка: кормление животных, медицинские осмотры и подготовку к выставкам. Также формируется отчёт по животным (количество, график кормления). Система спроектирована с учётом расширяемости — легко добавлять новые виды животных и типы сотрудников.

## Применение принципов SOLID

**SRP (Single Responsibility Principle)**  
Каждый класс решает одну задачу:  
- Animal — хранит данные о животном,  
- Zookeeper — кормит и убирает,  
- Veterinarian — проводит осмотры,  
- ZooReportGenerator — формирует отчёт.

**OCP (Open/Closed Principle)**  
Система открыта для расширения, но закрыта для изменения. Чтобы добавить новое животное (например, Turtle), достаточно создать новый класс, наследующий Animal. Никакой существующий код менять не нужно.

**LSP (Liskov Substitution Principle)**  
Любое животное (Lion, Parrot, Snake) можно использовать вместо родительского типа Animal. Программа работает корректно, независимо от конкретного вида.

**ISP (Interface Segregation Principle)**  
Интерфейсы разделены по функциям: Feedable, Cleanable, MedicalCheckable, Exhibitable. Каждый класс реализует только то, что ему нужно. Например, Snake не реализует Exhibitable, потому что не участвует в выставках.

**DIP (Dependency Inversion Principle)**  
Сотрудники (Zookeeper, Veterinarian) зависят от интерфейсов, а не от конкретных классов. Это позволяет легко заменять реализации и добавлять новые виды животных без изменения кода сотрудников.

## Возможные проблемы при расширении
- Если появятся сотрудники с комбинированными ролями (например, ветеринар + смотритель), может потребоваться рефакторинг.
- График кормления хранится как строка — сложно автоматизировать расчёты (например, «кормить чаще зимой»).
- Нет учёта состояния здоровья — нельзя автоматически отменять выставку для больного животного.

## Почему абстракции улучшили дизайн
- Код стал проще читать и поддерживать.
- Новые виды животных добавляются за несколько минут.
- Модули слабо связаны — изменения в одном месте не ломают другие.
- Легко писать тесты, так как зависимости чётко отделены.
